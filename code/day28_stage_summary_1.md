## 二、我已经真正掌握的 LLM 基础

下面这部分是 Day28 最核心的内容。这里不只是列概念，而是按“我已经能做什么”的角度来总结。

### 1. 我已经掌握了最小的 LLM 应用闭环

我已经理解并实践了一个最基础的 LLM 应用链路：

用户输入 -> 前端收集参数 -> 后端组织 Prompt -> 调用模型 -> 返回结构化结果 -> 前端展示

这件事看起来简单，但它其实是后面所有项目的底层模板。不管以后做总结助手、RAG、问答系统、工具调用，最底层都绕不开这个闭环。

这一点之所以算“真正掌握”，是因为我不是只看了架构图，而是已经把它做成了一个实际项目，并且能跑通。

我已经知道：

- 前端和模型之间通常不直接通信，而是通过后端中转。
- 后端不只是“转发请求”，还承担参数校验、Prompt 组织、异常处理、统一响应结构等职责。
- 统一响应格式会显著降低前端处理复杂度。

这意味着我已经从“会调用模型”进入到了“会做一个最小 LLM 产品”的阶段。

### 2. 我已经掌握了 Prompt 会直接影响结果质量

这一阶段里，我不是把 Prompt 当成一句随便写的提示，而是开始把它当成一个需要设计的“输入控制层”。

我已经理解并实践了这些点：

- Prompt 越模糊，模型越容易自由发挥。
- 当任务目标、格式约束、内容边界写得更清楚时，输出会更稳定。
- System Prompt 和 User Prompt 可以分层承担不同职责。
- 不同任务风格需要不同约束，而不是一个 Prompt 试图包打天下。

在项目里，这个认识已经具体落地为：

- `brief` 风格要求单段输出、限制字数、禁止引导语。
- `bullets` 风格要求 3～5 条、每条必须以 `- ` 开头、不能变成编号段落。
- `structured` 风格要求固定键名：`背景 / 关键动作 / 结果`。

也就是说，我已经不是在“向模型提问”，而是在“设计模型该如何回答”。

这是非常关键的一步，因为后面无论是 RAG 结果整理、结构化抽取、工具调用参数生成，Prompt 设计能力都会继续被用到。

### 3. 我已经掌握了 zero-shot 和 few-shot 的实际差别

通过 Day24 的实验，我已经不只是知道这两个词是什么意思，而是知道它们在真实任务里的区别。

我的实际认识是：

- zero-shot 上手快，适合先快速试任务。
- few-shot 更适合格式要求严格、判断标准需要统一的任务。
- 当我希望模型更稳定地输出 JSON、或减少判断标准漂移时，few-shot 通常更可靠。

这背后的核心不是“few-shot 一定更高级”，而是我已经知道什么时候需要给模型示例，什么时候不需要。

这说明我已经开始具备“根据任务复杂度调整 Prompt 策略”的意识，而不是把所有任务都用同一种写法处理。

### 4. 我已经掌握了上下文窗口和输入预算意识

Day22 的重点不是背定义，而是建立“预算”思维。

我已经理解：

- 模型不是无限接收文本的。
- 上下文窗口里要同时容纳 system prompt、用户输入、历史对话和模型输出。
- 输入越长，不只是可能报错，还会挤压输出空间、拉高时延、抬高成本。

更重要的是，我已经把这个理解和项目联系起来了：

- 长文本不能无脑全塞给总结助手。
- 如果输入太长，应该考虑限制长度、分段处理、二次汇总。
- 不是“输入越多越好”，而是“把预算留给最有价值的信息”。

这类意识在刚开始做小项目时很容易被忽略，但它恰好是后面做 RAG、长文总结、多轮对话时最重要的基础之一。

### 5. 我已经掌握了接口稳定性不是“可选项”

Day26 之后，我对“LLM 应用不是模型返回字就结束了”这件事有了更具体的认识。

我已经知道，一个能演示的接口和一个能稳定用的接口，中间差的往往不是模型能力，而是工程处理能力，包括：

- 参数校验
- 可读错误提示
- 全局异常捕获
- 请求日志
- 重试机制
- 请求耗时记录

这说明我已经开始从“模型实验”切换到“产品接口”视角。

这一步特别重要，因为真实用户不会关心你是不是调用了大模型，他们只会感知到：

- 为什么这次请求失败了
- 为什么有时候快、有时候慢
- 为什么报错时页面什么都没有

现在我已经能回答这些问题，并在代码里做第一层处理了。

### 6. 我已经掌握了成本意识的最小起点

Day27 的重点不是做精确账单系统，而是建立“每次调用都不是没有代价的”意识。

我已经开始记录：

- 输入长度
- 输出长度
- 请求耗时

虽然现在还是字符长度层面，还不是严格 token 统计，但它已经足够让我开始形成一些正确判断：

- 输入越长，通常请求越贵、越慢。
- 输出要求越详细，通常也会占更多预算。
- 很短的输入做总结时，压缩空间有限，输出长度可能和输入长度接近。
- 如果以后要做长文总结或多轮对话，必须更认真地管理上下文和输出约束。

这一步的意义在于：我已经不再把模型调用看成“黑盒魔法”，而是开始把它看成一种需要管理预算的资源。

### 7. 我已经掌握了前端在 LLM 产品里不只是“包一层页面”

因为这个阶段的目标本身就是“尽快把前端优势接进来”，所以我已经在项目里把前端真正作为产品体验的一部分接入了。

我已经在页面层面做到了：

- 输入区与结果区分离
- 风格切换
- loading 态
- 错误反馈
- 复制结果
- mock 模式与真实接口模式切换
- 接口调试面板
- 成本意识表展示

这说明我不是简单做了一个“能点按钮拿结果”的壳，而是已经开始理解：

- 前端要帮助用户理解系统当前状态
- 前端要把模型的不确定性翻译成更可感知的交互
- 前端是模型体验的一部分，不是最后才补的皮肤

这对后面的项目非常有价值，因为很多 AI 产品的差异，恰恰就来自“交互层有没有把模型能力用对”。

## 三、这一阶段最重要的收获，不是某个知识点，而是这些能力之间的关系

如果只把这一阶段拆成很多天来看，很容易觉得自己只是学了很多零散内容：

- 架构
- Prompt
- few-shot
- token
- 错误处理
- 日志
- 成本意识

但实际上，这些内容已经开始连成一条线。

这条线大概是这样的：

先做出一个最小 AI 应用 -> 再让输出更稳定 -> 再让接口更稳 -> 再开始关注成本和边界

这个顺序很重要，因为它符合真实项目的成长方式。

真实开发里，很少有人一开始就把“理论、工程、体验、成本”同时做到最好。更常见的路径是：

1. 先跑通。
2. 再变稳定。
3. 再变可维护。
4. 再变更省、更快、更适合扩展。

而我这一阶段其实正是在按这个顺序建立能力。

## 七、这一阶段还存在什么短板

为了让复盘更真实，这里也必须把不足写清楚。

### 1. 对 token 的理解还偏“概念级”

我已经知道 token 和上下文窗口的重要性，但目前更多还是字符级近似和概念级理解，还没有建立精确 token 统计与对比习惯。

### 2. 长文本处理还没有真正落地

我已经知道长文本不能全塞、应该做 chunking，但目前项目里还没有真正实现“分段总结 -> 汇总”的完整流程。

### 3. 稳定性还在基础阶段

虽然已经加了异常处理、日志和重试，但还没有做到：

- 分级错误码
- 更系统的监控
- 请求追踪面板持久化
- 自动化测试

### 4. 对成本的理解还比较粗粒度

现在已经能看输入长度、输出长度和耗时，但还没有建立：

- 不同任务的成本对比
- 不同模型的成本差异
- 成本和质量之间的量化权衡

这些都不是失败，而是下一阶段很自然的延伸方向。

## 八、这一阶段和后续项目的关系

Day28 不是为了“写一篇总结交差”，而是为了把接下来要走的路看清楚。

这一阶段学到的内容，后面至少会在三类项目里持续出现：

### 1. 在 RAG 项目里继续用

- 上下文窗口意识会直接决定检索片段怎么选、怎么截断、怎么汇总。
- Prompt 设计会直接影响引用格式、答案边界和幻觉控制。
- 成本意识会影响检索片段数量、上下文长度和模型选择。

### 2. 在多轮对话和 Agent 项目里继续用

- 历史对话如何保留，本质上还是上下文预算问题。
- 工具调用结果怎么组织给模型，本质上还是 Prompt 和结构化输入问题。
- 调用链变长后，稳定性、日志、重试会比现在更重要。

### 3. 在作品集项目里继续用

- 一个能展示的 AI 产品，不能只有“模型回答得出来”，还要能交互、能报错、能解释、能复盘。
- 前端体验、接口稳定性、成本意识，都会成为作品集区分度的一部分。

所以，这一阶段虽然是“基础”，但其实已经是后面所有复杂项目的骨架。
