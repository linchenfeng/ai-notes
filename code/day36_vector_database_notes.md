# Day 36：向量数据库基础与本地索引方案

## 这份文档要解决什么问题

Day36 的任务表面上是“学习向量数据库的基本概念，选一个轻量方案先上手”，但真正要解决的问题是：

当我们已经有了文档、切分脚本和 chunk 结果之后，接下来这些 chunk 的向量到底要放在哪里，后面又要怎么查。

如果这一步只停留在“听说过向量数据库”这种层面，后面做 embedding、建索引、相似度检索时会一直比较模糊。因为你会不知道：

- 为什么不能只把向量随便打印出来
- 为什么后面要有“索引环境”
- 本地最小版本到底应该长什么样
- 初学者现在是不是必须立刻上一个完整数据库

所以今天这份文档的目标，是把“向量数据库”和“本地轻量索引方案”讲清楚，并给当前项目选一个可继续推进的落地方案。

---

## 一、先用一句话理解向量数据库

向量数据库可以先粗略理解成：

一种专门用来存储向量，并支持按相似度快速检索的系统。

这里最关键的不是“数据库”这三个字，而是两个能力：

1. 能保存向量  
2. 能在用户提问时，快速找出最相近的向量  

如果把 RAG 里的文档 chunk 看成“要检索的知识单元”，那向量数据库或本地索引环境，本质上就是这些知识单元向量的存放和查找位置。

---

## 二、为什么 RAG 里需要“索引环境”

在 Day34 之后，我们已经有了 chunk，但 chunk 还只是文本。

后面 Day37 做 embedding 时，每个 chunk 会变成一组向量数字。问题来了：

- 这些向量放哪？
- 如何知道它对应哪个 chunk？
- 用户提问时怎么在一堆向量里快速找最相近的几条？

这就是“索引环境”存在的意义。

也就是说，它要解决的不是“生成答案”，而是“为检索阶段准备好可查找的知识结构”。

这一步至少要解决 3 类信息的组织问题：

### 1. 向量本身

每个 chunk 做 embedding 后，都会得到一个向量数组。

### 2. chunk 与向量的对应关系

不能只存向量，还要知道它对应：

- 哪个 `chunk_id`
- 哪篇文档
- 哪段正文
- 哪个标题

### 3. 检索方式

后面用户提问时，系统需要能根据“问题向量”去比较：

- 哪些 chunk 向量最相近
- top-k 应该返回哪些 chunk

所以索引环境不是可有可无的附属品，而是检索链路的核心基础设施。

---

## 三、向量数据库不一定一开始就要很重

这一步对初学者特别重要。

很多人一听到“向量数据库”，就会下意识以为必须立刻上：

- 很完整的向量数据库服务
- 复杂依赖
- 专门的服务器
- 很重的安装流程

但对当前这个项目阶段来说，没必要一开始就这么重。

因为我们现在的目标不是“做企业级大规模检索系统”，而是：

先做出一个能在本地稳定运行、能看懂、能继续扩展的最小 RAG 闭环。

所以 Day36 更合理的思路是：

先选一个轻量、本地、可读性强、便于调试的索引方案。

等整条链路跑通以后，再考虑是不是要升级成更成熟的向量数据库方案。

---

## 四、当前阶段可以考虑的几种方案

### 方案 1：本地 JSON / 文件索引

思路：

- chunk 元信息存成 JSON
- 后面 embedding 结果也先存成本地文件
- 检索时由 Python 脚本读进来比较

优点：

- 最轻量
- 没有额外服务依赖
- 最容易看懂和调试
- 很适合当前学习阶段

缺点：

- 数据量变大后效率一般
- 不适合特别大的知识库

### 方案 2：本地内存索引

思路：

- 运行时把所有向量读进内存
- 在 Python 进程里直接做相似度计算

优点：

- 实现简单
- 非常适合小规模实验

缺点：

- 进程结束后需要重新加载
- 不适合持久化要求更高的场景

### 方案 3：轻量向量数据库

例如后面可以考虑 Chroma、FAISS 等方向。

优点：

- 更接近真实项目结构
- 对检索和索引支持更完整

缺点：

- 学习曲线和安装复杂度更高
- 现在还没到必须上它的时候

## 五、我给当前项目选择的轻量方案

结合你当前的学习阶段和现有代码基础，我认为最合适的 Day36 方案是：

先用“本地文件索引 + 本地目录结构”的轻量方案上手。

也就是：

- 先把索引环境目录搭起来
- 把后续要保存的内容位置约定好
- 让 Day37、Day38、Day39 都能继续往这个目录里接

这个方案为什么最合适：

### 1. 它和当前阶段目标匹配

现在最重要的是把 RAG 整条链路跑通，而不是一开始就上复杂基础设施。

### 2. 它最适合学习

本地文件可见、结构清楚，后面你一眼就能知道：

- chunk 放哪了
- embedding 放哪了
- 索引配置放哪了

### 3. 它方便调试

如果后面检索效果不好，你可以很容易回头检查：

- 读取有没有问题
- 切分结果是什么
- embedding 是否对上 chunk

### 4. 它后续还能升级

等后面最小闭环稳定后，再切换到更完整的向量数据库，也不会推翻前面的理解。

## 六、当前项目的本地索引环境设计

我给当前项目先建立的本地索引目录是：

[day36_local_index](/Volumes/学习资料/ai-notes/code/day36_local_index)

这个目录的定位不是“今天就塞满数据”，而是先把后面索引相关文件的落点明确下来。

建议结构如下：

```text
day36_local_index/
├── README.md
├── index_config.json
├── chunks.json
├── embeddings.json
└── search_results.json
```

含义分别是：

- `README.md`
说明这个目录是做什么的

- `index_config.json`
记录当前索引方案的配置和约定

- `chunks.json`
后续可以保存切分结果

- `embeddings.json`
后续可以保存 chunk 对应向量

- `search_results.json`
后续调试检索时可以临时输出结果

这是一种“先搭骨架，再逐步填内容”的方式。

## 七、为什么今天不直接上完整向量数据库

这一步我想特别说清楚，因为很多人会误以为“现在不用完整向量数据库是不是太简陋了”。

我的判断是：不简陋，反而更适合当前节奏。

原因有 4 个：

### 1. 当前数据规模很小

现在资料库只有 20 篇左右文档，切出来的 chunk 数量也不大。

这种规模下，先用本地索引完全够支撑学习和验证。

### 2. 当前更重要的是理解流程

你现在最需要搞清楚的是：

- chunk 是什么
- embedding 之后如何对应 chunk
- 检索到底在比什么

这些理解先建立起来，比“工具选得多高级”更重要。

### 3. 过早引入复杂依赖会干扰主线

如果现在立刻上更重的数据库工具，容易把注意力转移到：

- 安装
- 兼容性
- 环境配置
- API 细节

而不是 RAG 主线本身。

### 4. 本地索引环境已经足够支撑后续几天

Day37～Day39 的最小目标，其实完全可以先在本地轻量索引上跑通：

- 先做 embedding
- 先存起来
- 先做简单相似度比较
- 先返回 top-k chunk

等真正跑通后，再考虑是不是升级存储方案。

## 九、今天的结论

如果用最直接的话总结 Day36，我会这样说：

向量数据库的核心价值，不在“数据库”这个名字，而在“能不能把向量和它对应的知识单元组织起来，并支持后续相似度检索”。
