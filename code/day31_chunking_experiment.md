# Day 31：Chunking 切分实验

## 这份实验要解决什么问题

Day31 的任务不是只知道“chunking 是切分文档”这么一句话，而是要真正看见：

同一篇长文，用不同方式切分之后，最后得到的 chunk 会很不一样，而这些差异会直接影响后面的检索质量。

如果这一步没有具体实验，chunking 很容易停留在抽象理解里。可一旦进入真正的 RAG 项目，你很快就会发现：

- 切得太大，相关信息会被无关内容稀释
- 切得太碎，上下文会断裂
- 切得不按结构走，标题和正文可能被拆散
- 不做 overlap，跨段信息可能丢失

所以今天这份实验的目标，是用一篇现成长文，按 3 种方式切分，然后对比它们的差别、优点、风险和适用场景。

---

## 一、实验对象

本次实验选用的长文来源是：

[day28_stage_summary_1.md](/Volumes/学习资料/ai-notes/code/day28_stage_summary_1.md)

选择它的原因有三个：

1. 它足够长，天然适合做 chunking 对比  
2. 它内部有清晰标题和分段，便于观察不同切法的效果  
3. 内容本身是学习总结，语义密度较高，比较接近以后知识库问答里常见的文档类型  

为了方便展示，这次实验不追求“把整篇文章完整切完并全部列出”，而是重点展示：

- 同一篇文档在不同切分策略下，前几段 chunk 会长什么样
- 每种切分方式的思路是什么
- 它们会给后续检索带来什么影响

## 二、实验目标

本次实验希望回答三个具体问题：

1. 固定长度切分、按段落切分、按标题结构切分，结果到底有什么差异  
2. 哪种切法更容易保持语义完整  
3. 哪种切法更适合后面的知识库问答场景

## 三、先给出原始文档特征判断

在切分之前，先判断原始文档长什么样，这会决定什么切法更合理。

这篇文档的特点是：

- Markdown 结构明显
- 有很多一级、二级标题
- 每个标题下面通常是成组解释
- 有列表、总结、结论和反思
- 单段内容有长有短

这说明它不是一篇完全平铺直叙的纯正文，而是一篇“结构化强、层级清楚”的文档。

因此，从直觉上看：

- 如果只按固定字数切，容易打断一个完整论点
- 如果按自然段切，会比固定字数更自然
- 如果按标题层级切，通常最能保留知识结构

但具体效果，还是要通过实际切分来看。

## 四、切分方式一：按固定长度切分

### 方法说明

这是一种最朴素的切法。

思路是：

不太关心文档结构，只按固定字数窗口去切，比如每 300～500 字切成一个 chunk。

这种方法的优点是实现简单，适合快速起步，也最容易写成稳定脚本。

但它的问题也很明显：

- 容易把一句完整的话切断
- 容易把标题和正文拆开
- 一个 chunk 里可能混入多个主题

### 本次实验的模拟规则

- 目标：每个 chunk 大约 350 字左右
- 不额外做 overlap
- 仅展示前 3 个示例 chunk

### 示例 chunk

#### Chunk A1

```text

## 这份总结要回答什么

Day28 不是再补一个零散笔记，而是要回答一个更关键的问题：

我到现在为止，已经真正掌握了哪些 LLM 基础，而且这些基础是不是已经通过项目被验证过。

这一阶段的主题不是“学很多术语”，而是“先做出第一个可运行、可展示、可联调的 AI Web 小项目”，然后在这个项目上继续补齐 prompt、稳定性、成本意识这些真实开发里会遇到的基础能力。
```

## 五、切分方式二：按自然段落切分

### 方法说明

这种切法比固定长度更“尊重原文表达单位”。

思路是：

以自然段为基础，一个 chunk 可以由 1～3 个连续段落组成，尽量不把段内内容截断。

这样做的好处是：

- 语义通常更完整
- 读起来更自然
- 不容易把一句话从中间切开

但它也有局限：

- 有些段落太长，仍然可能过大
- 有些段落太短，信息量可能不足
- 如果不控制段落组合规则，chunk 大小会不均匀

### 本次实验的模拟规则

- 以自然段为基本单位
- 每个 chunk 尽量由 2～3 个相邻自然段组成
- 标题尽量跟后续段落放在一起

### 观察

这种切法比固定长度更自然，但也暴露出一个新问题：

- 有时一个标题和它后面的解释仍然比较短，单独成 chunk 后信息密度不够

## 六、切分方式三：按标题结构切分

### 方法说明

这是一种更适合结构化 Markdown 文档的做法。

思路是：

优先按照标题层级来切，把一个标题及其下属解释尽可能保留在同一个 chunk 里。

例如：

- `## 一、这一阶段我完成了什么` 下面的内容尽量放在一起
- `### 1. 我已经掌握了最小的 LLM 应用闭环` 下面的解释尽量不要拆散

这种方法的核心优点是：

- 能保留知识结构
- 检索结果更容易和“一个明确主题”对应
- 很适合后面做引用来源

它的缺点是：

- 如果某个标题下面内容过长，还需要进一步二次切分
- 对结构不清晰的文档不一定适用
- 实现上通常比固定长度切分更复杂

### 本次实验的模拟规则

- 以 Markdown 标题为主边界
- 保留标题与说明内容的关联
- 如果某个小节明显过长，再考虑在小节内部细切

### 观察

这种切法的优点最明显：

- chunk 的主题边界最清楚  
- 标题和正文天然绑定，检索结果更可解释  
- 后面如果要做引用，会更容易告诉用户“这段来自哪个主题”  

但它也不是完美的：

- 如果某个标题下面内容很长，一个 chunk 可能仍然过大  
- 需要依赖文档本身有比较好的标题结构  
- 对纯聊天记录、杂乱网页文本、扫描 OCR 文档不一定适用  

### 初步结论

对于这类结构清晰的 Markdown 学习文档，按标题结构切分通常是最有信息组织优势的方案。

## 七、三种切分方式的对比总结

| 维度 | 固定长度切分 | 按自然段切分 | 按标题结构切分 |
| --- | --- | --- | --- |
| 实现难度 | 最低 | 低到中等 | 中等 |
| 对原文结构依赖 | 低 | 中等 | 高 |
| 语义完整性 | 一般 | 较好 | 通常最好 |
| chunk 大小均匀性 | 最均匀 | 不均匀 | 不均匀 |
| 是否容易切断上下文 | 高 | 中等 | 较低 |
| 是否适合 Markdown 文档 | 一般 | 较好 | 很适合 |
| 是否适合后续引用来源 | 一般 | 较好 | 最好 |

从这次实验看，可以得到一个很实用的判断：

- 如果只是想快速起步，固定长度切分最容易做  
- 如果文档是普通长文，按自然段切通常更自然  
- 如果文档本身层级清楚，按标题结构切分通常最适合知识库问答

## 八、这次实验最关键的收获

这次实验最重要的收获不是“学会了 3 种切法”，而是开始真正理解：

chunk 不是随便切一刀就行，chunk 的边界其实决定了后面“检索到的是什么单位”。

你后面检索的不是整篇文章，而是 chunk。

所以如果 chunk 本身有问题，后面整条链路都会受影响：

- chunk 太杂，检索到了也不聚焦  
- chunk 太碎，检索到了也解释不完整  
- chunk 没有结构，后面引用来源会变得很难看  

也就是说，chunking 不是一个小预处理细节，而是 RAG 质量的核心环节之一。

## 九、如果把这次实验放回后面的 RAG 实现里

Day31 的实验会直接影响后面这些实现选择：

### 对 Day33 读取脚本的影响

读文档时就要尽量保留：

- 标题
- 段落
- 文件名
- 路径

因为这些信息会影响切分质量。

### 对 Day34 切分脚本的影响

写切分脚本时，不应该只做一种固定切法，而应该至少保留后续可调整空间，例如：

- chunk size
- overlap
- 按段落切还是按标题切

### 对 Day39 检索器的影响

后面检索效果不好时，不能只怀疑 embedding 模型，也要优先排查：

- chunk 切得对不对
- chunk 边界合不合理
- 一个 chunk 里是不是混了太多主题

### 对 Day41 引用来源的影响

如果 chunk 保留了标题结构，后面做来源引用会明显更自然，因为你更容易告诉用户：

- 这段内容来自哪个章节
- 它在整篇文档里属于什么主题

## 十、我现在对 chunking 的理解

如果要用自己的话来概括，我现在会这样理解 chunking：

chunking 不是简单把长文切小，而是把长文切成“既适合检索、又尽量保留语义完整性”的知识单元。

这句话里最关键的是“知识单元”。

因为在 RAG 里，真正被 embedding、被索引、被检索、被送进生成模型的，不是整篇文档，而是这些 chunk。

所以 chunk 的质量，几乎就是后面 RAG 质量的地基。

## 十一、今天这一步会在后面的哪个项目里派上用场

今天学 chunking，是为了后面做知识库问答时，让系统不是机械地把文档切碎，而是能把资料切成更适合检索和回答的片段。

以后无论你做：

- 学习笔记知识库
- 技术文档问答
- FAQ 助手
- 公司内部知识库

都绕不开 chunking。

因为只要资料稍微多一点，系统最终检索的就不是“整篇文档”，而是“切分后的 chunk”。
