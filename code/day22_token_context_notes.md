# Day 22 术语笔记：Token 与上下文窗口

## 1. 什么是 Token
- Token 不是“字符数”，也不是“单词数”。
- 模型在内部处理文本时，会先把文本切成更小单位，这些单位就是 token。
- 同一段中文、英文、数字、符号，在不同模型里切分方式可能不同。

## 2. 什么是上下文窗口（Context Window）
- 上下文窗口可以理解成：模型一次请求里“能看到的总信息上限”。
- 这个总信息包含：
  - system prompt
  - 用户输入
  - 历史对话
  - 模型要输出的内容（输出 token 也占预算）

## 3. 为什么长文本不能无脑全塞
1. 会超出模型上限
- 一旦总 token 超过窗口，接口会报错，或者被截断。

2. 会挤压输出空间
- 输入塞太满时，留给模型生成结果的 token 变少，输出容易被截断。

3. 成本和时延上升
- 输入越长，请求越慢，API 成本通常也越高。

4. 信息密度反而下降
- 把大量低价值内容一起塞给模型，可能稀释关键信息，结果不稳定。

## 4. 在你这个 AI 总结助手里的实际影响
- 当用户粘贴超长文章时：
  - 可能直接失败（超限）
  - 可能输出不完整（被截断）
  - 可能响应很慢（体验差）

## 5. 常见应对策略
1. 先做长度限制
- 比如先限制输入字符数，超过就提示用户分段。

2. 分段总结（Chunking）
- 先把长文切成多段，逐段总结，再做二次汇总。

3. 预留输出预算
- 不要把窗口塞满，给模型留足够输出空间。

4. 只保留必要上下文
- 对多轮对话只保留最近几轮，避免历史无限增长。

## 6. 一句话记忆
上下文窗口是预算，不是越满越好；要把预算花在最关键的信息上。

## 7. 与后续项目的关系

- Day22 的核心价值就是建立“预算意识”：输入、历史、输出都在争夺同一窗口资源。
