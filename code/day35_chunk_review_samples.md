# Day 35：切分样本复盘

## 这份复盘要解决什么问题

Day35 的重点不是再写一个新脚本，而是回头检查：

昨天 Day34 切出来的 chunk，到底合不合理。

这里的“合理”，不是只看代码能不能跑，而是要看这些 chunk 是否真的适合作为后续 RAG 的知识单元。

因为后面做 embedding、建立索引、检索和生成时，系统真正处理的不是整篇文档，而是这些 chunk。

所以如果 chunk 本身质量不行，后面再怎么调 embedding、top-k、Prompt，效果都会受到限制。

这份复盘主要回答四个问题：

1. 当前切分结果整体表现怎么样  
2. 哪些样本切得比较合理  
3. 哪些样本暴露了问题  
4. 下一步该优先优化什么  

---

## 一、复盘对象

本次复盘基于昨天的切分脚本：

[day34_split_docs.py](/Volumes/学习资料/ai-notes/code/day34_split_docs.py)

当前切分策略是：

- 固定字符窗口切分
- `chunk_size = 300`
- `overlap = 50`

实际运行结果：

- 成功加载 21 篇文档
- 成功切分得到 24 个 chunk

也就是说，大部分短文档只产生了 1 个 chunk，只有较长文档被切成了多个 chunk。

---

## 二、我这次判断 chunk 是否合理的标准

为了避免复盘变成主观感受，这里先明确判断标准。

我主要用 5 个标准来看每个 chunk：

### 1. 主题是否清楚

一个 chunk 最好能围绕一个相对明确的主题，而不是把多个不太相关的内容拼在一起。

### 2. 语义是否完整

读这个 chunk 时，最好能看懂它在讲什么，而不是总感觉“像是从中间被截断了”。

### 3. 是否保留了足够上下文

chunk 不一定要完整覆盖整篇逻辑，但至少要保留足够的信息，使它在被单独检索出来时仍然有价值。

### 4. 是否适合后续检索

如果用户提一个问题，这个 chunk 是否有机会作为“候选证据”被准确召回，而不是因为过杂或过碎导致价值下降。

### 5. 是否方便后续做引用来源

如果后面要把这个 chunk 展示给用户，或者告诉用户“答案来自哪里”，这个 chunk 是否具有足够明确的来源身份和内容边界。

---

## 三、整体结论先说

先给出整体判断：

当前这版切分结果，作为 Day34 的最小可运行版本是合格的，但作为后续真正想追求更稳检索质量的方案，还不够理想。

为什么说它“合格”：

- 能稳定运行
- 能批量把文档切成 chunk
- 能保留来源路径、字符区间和 chunk ID
- overlap 机制已经在发挥作用
- 对大部分短文档来说，单 chunk 结果基本可读

为什么说它“还不够理想”：

- 对结构化文档，固定长度切分仍然容易切断主题边界
- 有些 chunk 只是清单中间片段，可检索价值不高
- 有些长 README 类型文档被切成多个列表残片，语义完整性一般
- 当前策略没有显式利用标题结构和段落结构

所以 Day35 的结论不是“昨天做错了”，而是：

昨天做的是正确的第一步，但现在已经能明确看出下一步该往哪里优化。

## 四、切得比较合理的样本

下面先看一些“当前策略下效果还不错”的例子。

### 样本 1：`doc01_llm_basics_chunk_000`

来源文档：

[doc01_llm_basics.md](/Volumes/学习资料/ai-notes/code/docs/doc01_llm_basics.md)

当前 chunk 特点：

- 标题完整保留
- 三段正文都保留在同一个 chunk 中
- 主题集中在“大语言模型基础”
- 没有出现明显硬截断

为什么说它合理：

因为这篇文档本身就比较短，字符数没有超过当前窗口上限，所以切分后仍然保持了完整性。

如果后续用户问：

- “什么是大语言模型？”
- “为什么初学者先做最小闭环？”

这个 chunk 都很有机会作为一个较完整的候选证据。

这里说明一件很重要的事：

固定长度切分不是完全不好。对于短文档，它往往能天然得到不错结果。

### 样本 2：`doc04_embedding_basics_chunk_000`

来源文档：

[doc04_embedding_basics.md](/Volumes/学习资料/ai-notes/code/docs/doc04_embedding_basics.md)

当前 chunk 特点：

- 标题和解释内容都保留下来了
- 主题聚焦在 embedding 的基本概念
- 内容量适中，没有明显碎裂

为什么说它合理：

因为这类概念型短文档，本身篇幅适中，固定长度切分几乎等价于“整篇保留”。

这对后续 RAG 非常友好：

- embedding 结果会对应比较纯净的主题
- 检索出来时也更容易直接用于回答

### 样本 3：`doc19_search_vs_generation_chunk_000`

来源文档：

[doc19_search_vs_generation.md](/Volumes/学习资料/ai-notes/code/docs/doc19_search_vs_generation.md)

当前 chunk 特点：

- 主题边界清晰
- 内容聚焦于“检索与生成的分工”
- 单独拿出来时也有完整表达

为什么说它合理：

因为它很像后续理想 chunk 的样子：

- 一块内容只讲一件事
- 读者不需要依赖大量前文也能理解
- 很适合作为问答证据

这说明当前资料库中那些“短而主题单一”的文档，在现有切分策略下表现是不错的。

## 五、暴露问题最明显的样本

下面开始看更有价值的部分：哪些 chunk 暴露出了当前切分策略的局限。

### 样本 4：`README_chunk_001`

来源文档：

[docs/README.md](/Volumes/学习资料/ai-notes/code/docs/README.md)

当前 chunk 主要内容：

- 文档清单中间部分
- 文件名列表片段
- 列表项之间被截断

为什么它不够理想：

这个 chunk 的内容虽然“没有错误”，但它作为知识单元的价值很低。

主要问题是：

- 它更像目录残片，不像主题内容块
- 如果检索出来，用户很难直接从中获得可回答问题的证据
- overlap 虽然让前后文本多了衔接，但并没有真正解决“内容本身是清单碎片”的问题

这说明固定长度切分对“列表型文档、目录型文档”不够友好。

### 样本 5：`README_chunk_002`

来源文档：

[docs/README.md](/Volumes/学习资料/ai-notes/code/docs/README.md)

当前 chunk 主要内容：

- 文档清单的后半部分
- 从一个文件名中间开始
- 结尾才逐渐进入“用途”部分

为什么这是一个很典型的问题样本：

它把两类不同性质的信息拼在了一起：

- 前半段是目录清单残片
- 后半段开始进入说明文字

也就是说，这个 chunk 的主题边界不干净。

如果后续检索问题和“用途”有关，这个 chunk 有可能被召回；
但它前半段的大量目录碎片会稀释信息密度。

这正是固定长度切分最典型的问题之一：

并不是每个 300 字窗口都天然对应一个合理知识单元。

### 样本 6：长文档跨边界切分问题

当前策略虽然加了 overlap，但它仍然会遇到一种情况：

如果一个完整主题刚好跨在 300 字边界附近，那么：

- 前一个 chunk 会带一点后文
- 后一个 chunk 会带一点前文

但这不等于“主题就真的完整保住了”。

它只是降低了断裂程度，不是彻底解决了结构问题。

也就是说：

overlap 是缓冲，不是根治。

这也是为什么昨天的实现虽然合理，但今天复盘后仍然能明确看出：

只靠固定长度 + overlap，还不够。

## 六、为什么 README 类文档在这次切分里最容易出问题

这次最明显的问题集中在 `docs/README.md`，不是偶然。

因为 README 这类文档本身具有几个特征：

- 有目录列表
- 有编号清单
- 有分节说明
- 信息类型混合

对这种文档来说，如果只按字符窗口切：

- 很容易从列表中间切开
- 很容易让文件名残片单独成为一个 chunk
- 很容易把“清单内容”和“说明内容”拼到一起

这说明一个更深层的事实：

chunking 策略要和文档类型匹配。

并不是同一种切法适合所有文档。

对于 README 这种结构明显的 Markdown，按标题结构切通常会更合理。

## 七、从这次结果看，当前策略最适合什么，不适合什么

### 当前策略比较适合

- 篇幅较短的知识卡片型文档
- 每篇文档只围绕一个主题的小型说明文
- 先快速跑通 RAG 最小闭环的实验阶段

### 当前策略不太适合

- 结构复杂的 Markdown 长文
- 带大量列表、目录、编号的说明文档
- 希望后续引用来源非常自然的场景
- 希望 chunk 与知识主题边界高度一致的场景

这说明 Day34 的实现很适合作为“第一版上游处理”，但如果后面想让检索质量更稳，切分策略还需要继续进化。

## 八、如果把这次复盘变成后续优化建议

根据这次样本观察，后续最值得优先做的优化有 3 个。

### 1. 增加按段落切分能力

原因：

很多 Markdown 或说明类文档，其实天然就按段落组织信息。

如果先按段落切，再控制 chunk 大小，通常会比单纯按字符窗口更自然。

### 2. 增加按标题结构切分能力

原因：

当前资料库里大部分文档都是 Markdown，标题结构很明显。

如果切分时显式利用标题：

- chunk 主题边界会更清楚
- 检索到的内容更容易直接回答问题
- 后续做引用来源会更自然

### 3. 对不同文档类型使用不同切分策略

原因：

README、概念卡片、长篇总结文档，它们不一定适合同一种切法。

未来可以考虑：

- 短文档：整篇保留
- 普通长文：按段落切
- Markdown 结构文：按标题切

这会比“一刀切的固定长度”更接近真实 RAG 项目的做法。
