# Day 15 项目说明文档

## 1. 项目题目
AI 总结助手（Web 版）

## 2. 选题原因
- 与前面 Day 11（结构化输出）和 Day 12（Prompt 对比）衔接自然，复用价值高。
- 功能边界清晰，适合做第一个前后端联调项目。
- 结果可视化强，便于做截图和作品集展示。

## 3. 项目目标（一句话）
用户输入一段文章后，网页能返回一段高质量总结，并支持不同总结风格切换。

## 4. 目标用户
- 需要快速阅读信息的学生和职场用户。
- 希望减少阅读时间的人。
- 初学 AI 应用开发、需要可展示项目的开发者（你自己）。

## 5. MVP 功能清单（最小可用版本）
- 文本输入框：粘贴原文内容。
- 总结按钮：触发总结请求。
- 结果展示区：展示模型输出。
- 风格选择：`简洁版`、`要点版`、`结构化版`。
- 基础状态提示：`加载中`、`失败提示`。

## 6. 非功能需求（第一版）
- 单次请求响应时间：目标 3~10 秒（本地模型视机器性能波动）。
- 输入长度建议：先限制在 2000 字以内，避免过长导致体验下降。
- 错误可见：前端必须展示可理解的错误信息，不允许无响应。

## 7. 页面草图（文字版）
- 顶部：项目标题 + 简短说明。
- 中间左侧：输入区（多行文本框）。
- 中间右侧：输出区（总结结果）。
- 输入区下方：风格下拉框 + 生成按钮。
- 输出区下方：复制按钮（Day 20 可补）。

## 8. 前后端职责划分（为 Day 16/18 做准备）
- 前端（React/Next.js）：
  - 收集输入文本和风格选项。
  - 调用后端接口。
  - 处理加载态、错误态、结果态。
- 后端（FastAPI）：
  - 接收文本与风格参数。
  - 组装 Prompt 调用本地 Ollama。
  - 返回统一 JSON 格式给前端。

## 9. 接口草案（初稿）
- 路径：`POST /generate-summary`
- 请求体：
```json
{
  "text": "用户输入原文",
  "style": "brief"
}
```
- 响应体（成功）：
```json
{
  "ok": true,
  "summary": "模型生成的总结",
  "model": "qwen2.5:0.5b"
}
```
- 响应体（失败）：
```json
{
  "ok": false,
  "error": "错误原因说明"
}
```

## 10. 风格定义（第一版）
- `brief`：1 段话，80~120 字。
- `bullets`：3~5 条要点列表。
- `structured`：固定字段输出（背景/关键动作/结果）。

## 11. 验收标准（Day 19 前）
- 页面可打开并可输入文本。
- 点击按钮后能看到模型返回内容。
- 至少 1 种风格稳定可用。
- 出错时有可读提示。

2. Day 17 用 mock 数据先打通前端页面。
3. Day 18 再接入本地 Ollama 的真实后端接口。
4. Day 19 完成前后端联调。
