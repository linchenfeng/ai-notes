# 功能增改流程

这份文档用于约束在 `../bms` 中做真实业务功能增改时的默认工作流。目标不是给出唯一模板，而是帮助 Codex 先定位、后模仿、再落点。

## 1. 先定位业务切片

- 先确定需求属于哪个业务目录，例如 `member/project/query`、`member/project/apply`、`member/myinfo`。
- 优先沿用现有业务树，不要因为当前需求而新建顶层业务目录。
- 如果需求横跨多个页面，先找出共同的业务归属，再决定是否拆分到多个文件。

## 2. 先看相邻代码

至少检查这些位置是否已有兄弟实现：

- `src/pages`
- `src/router`
- `src/api`
- `src/interfaces`
- `src/store`
- `src/plugins`
- `src/utils`

优先看同层级、同业务目录、同交互类型的文件，而不是全局搜索最远处的示例。

## 3. 决定四层是否要一起补齐

把下面四层视为功能增改的核心落点：

- `src/pages`
- `src/router`
- `src/api`
- `src/interfaces`

默认先判断：

- 是只改页面展示，还是连路由入口都要动
- 是只调已有接口，还是需要新增 API 封装
- 是只复用现有类型，还是需要新增请求或响应类型

不要只在一层里硬塞逻辑，导致同业务切片在别处失配。

## 4. 按最小闭环落点

常见闭环不是“只写一个页面”，而是下面几类：

- 页面 + 路由
- 页面 + API + 类型
- 页面 + 路由 + API + 类型
- 页面 + 局部组件 + API + 类型

只有当状态天然属于跨页面共享时，才考虑补 `store`。

## 5. 复用优先级

新增代码前按这个顺序检查：

1. 同目录兄弟页面是否已有相同模式
2. 当前业务目录下是否已有局部组件可复用
3. `src/plugins` 中是否已有全局能力匹配
4. `src/utils`、`src/hooks` 中是否已有工具或 hooks 可复用
5. `src/store` 中是否已有共享状态可接入

只有前面都不合适时，才新增抽象。

## 6. 提交前自检

- 文件落点是否与现有业务目录平行对齐
- 命名是否复用项目内现有业务术语
- 页面结构、路由 meta、API 调用、类型命名是否与兄弟文件一致
- 是否避免了新依赖、新请求层、新 UI 抽象层
- 是否用最轻量的方式完成了校验
