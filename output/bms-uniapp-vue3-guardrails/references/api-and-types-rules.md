# API 与类型规范

这份文档用于约束 `../bms/src/api` 与 `../bms/src/interfaces` 的新增和修改，重点是业务目录对齐、请求层统一和类型落点清晰。

## API 规范

- 统一复用 `src/api/api.request.ts` 中的 `request<T>`
- 不引入新的 axios、fetch 二次封装或页面内临时请求方式
- API 文件优先落到与业务一致的目录中
- 命名优先沿用现有 `api.xxx.ts` 风格和同业务目录中的命名方式

## 类型规范

- 请求与响应类型优先放到 `src/interfaces` 对应业务目录下
- 先检查相邻目录中是否已有同类分页、列表、详情、表单类型
- 优先复用现有业务术语、字段名和结构含义
- 不为了表面整洁而随意重命名后端字段

## 落点对齐

新增接口或类型时，优先判断这几层是否应该同步补齐：

- `src/pages/<业务目录>`
- `src/router/<业务目录>`
- `src/api/<业务目录>`
- `src/interfaces/<业务目录>`

落点重点不是固定某个目录名字，而是保持同业务切片在四层结构中的平行映射。

## 后续建议继续细化

- 列表查询参数与分页返回类型规范
- 详情接口与详情模型规范
- 表单提交参数规范
- 文件上传与文件预览接口规范
- 公共响应结构与错误处理规范
