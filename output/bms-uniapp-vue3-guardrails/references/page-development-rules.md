# 页面开发规范

这份文档用于约束 `../bms/src/pages` 下页面的新增与修改，重点是页面骨架、局部组件边界和与相邻页面保持一致。

如果任务明确是申请页，优先继续阅读 `apply-page-rules.md`。
如果任务明确是详情页，优先继续阅读 `detail-page-rules.md`。

## 页面开发的第一原则

- 先找相同业务目录下的兄弟页面
- 先模仿真实相邻页面，再决定自己的结构
- 不直接套通用 Vue 页面模板

## 页面目录落点

- 页面放到 `src/pages` 下已有的业务路径中
- 如果功能属于 `member/project/query/foo`，优先在同层级页面附近落点，而不是新起一套目录
- 页面私有组件优先放到当前页面或当前业务目录下的 `components/`
- 只有跨多个业务切片稳定复用时，才考虑提升为全局复用组件

## 页面结构规范

优先从相邻页面抽取结构，而不是自创骨架。沉淀时重点关注这几类页面：

- 列表页
- 详情页
- 表单页
- 流程页
- 信息汇总页

建议关注这些结构点是否在相邻页面中已有共识：

- 页面头部与标题区
- 筛选区、操作区、列表区、分页区
- 分组卡片、折叠区、附件区、审批流区
- 底部操作栏、悬浮操作、空态、异常态

## 页面脚本风格

- 沿用相邻文件已有的 `<script setup lang="ts">` 组织方式
- 如果相邻页面同时保留简短的 `export default { name }`，就继续沿用
- 如果相邻页面使用 `@dcloudio/uni-app` 生命周期，例如 `onLoad`、`onShow`，就优先复用同样的生命周期入口
- 不为了“更现代”而强行把相邻页面改成另一套风格

## 页面内组件复用

在新建局部组件前，先检查是否已有这些能力：

- 导航栏
- tab 切换
- 上传
- 预览
- tooltip
- info-card
- 数字输入
- 国家选择

如果当前需求只是现有能力的业务包装，优先轻量组合，不要重新发明基础组件。

## 页面样式规范

- 优先沿用相邻页面的样式组织方式
- 默认使用 `lang="scss"`
- 除非相邻文件另有约定，优先使用 `scoped`
- 移动端布局优先沿用项目内已使用的 `rpx` 风格
- 颜色、间距、圆角、卡片、分栏等视觉模式尽量向兄弟页面靠齐
